# PseudoForge Corpus Q&A

## Purpose

Use PseudoForge batch artifacts as the source of truth for answering questions about a binary. Do not require IDA, Hex-Rays, or the PseudoForge runtime unless the user explicitly asks to regenerate artifacts.

Prefer focused evidence over loading the whole corpus. For every important claim, cite the function EA, function name, and artifact path.

## Expected Inputs

Accept any of these:

- A PseudoForge IDA CLI output directory.
- `pseudoforge-corpus-index.json`.
- `pseudoforge-corpus-metadata.json`.
- `pseudoforge-corpus-overview.md`.
- A `functions/<ea>_<function>/` export bundle.
- A focused `qa-context.md` generated by `pseudoforge_corpus_qa.py`.
- A `.forge` aggregate file or JSONL batch report when the index is unavailable.

Preferred full handoff set:

- `pseudoforge-corpus-index.json`
- `pseudoforge-corpus-metadata.json`
- `pseudoforge-corpus-overview.md`
- `functions/<ea>_<function>/*.ida-batch-summary.json`
- `functions/<ea>_<function>/*.cleaned.cpp`
- `functions/<ea>_<function>/*.raw.cpp`
- `functions/<ea>_<function>/*.raw-vs-cleaned.diff`
- `functions/<ea>_<function>/*.rename-map.json`
- `functions/<ea>_<function>/*.rule-report.json`
- `functions/<ea>_<function>/*.buffer-contracts.json`
- `functions/<ea>_<function>/*.warnings.json`

## Artifact Placement

Keep the skill separate from the corpus artifacts.

Recommended skill location for Codex:

```text
%USERPROFILE%\.codex\skills\pseudoforge-corpus-qa\SKILL.md
```

Recommended long-term corpus location:

```text
F:\pseudoforge-corpora\<target-name>\
```

Recommended repo-local temporary corpus location:

```text
F:\kernullist\PseudoForge\pseudoforge_out\<target-name>\
```

The repo-local `pseudoforge_out/` directory is ignored by Git in PseudoForge, so it is suitable for smoke runs and local validation artifacts. For long-term binary research, prefer `F:\pseudoforge-corpora\<target-name>\` or another external corpus root.

Do not place large binary corpus outputs inside the skill folder. A skill should contain instructions and small references; the corpus root should contain generated analysis artifacts.

Example IDA corpus generation:

```powershell
python -B .\tools\pseudoforge_ida_cli.py `
  "C:\IDA\ida64.exe" `
  "D:\targets\driver-a.i64" `
  "F:\pseudoforge-corpora\driver-a"
```

Example prompt to another AI agent:

```text
Use the pseudoforge-corpus-qa skill.

Corpus root:
F:\pseudoforge-corpora\driver-a

Question:
Explain this driver's IOCTL dispatch structure and main callback registration flow.
```

Example prompt for a repo-local temporary corpus:

```text
Use the pseudoforge-corpus-qa skill.

Corpus root:
F:\kernullist\PseudoForge\pseudoforge_out\driver-a

Question:
이 드라이버의 전체 구조를 entrypoint, dispatch, callback, cleanup 흐름으로 나눠서 설명해줘.
```

## Core Workflow

1. Locate the corpus root.
   - If the user provides a directory, look for `pseudoforge-corpus-index.json` first.
   - If only an index path is provided, treat its parent directory as the corpus root.
   - If only `qa-context.md` is provided, answer from that context and say when broader corpus evidence is unavailable.

2. Check corpus completeness.
   - Read `pseudoforge-ida-run.json` when present.
   - Check `status`, `summary`, `status_counts`, `llm_status_counts`, `functions_dir`, `report_path`, and corpus artifact paths.
   - If the run failed or is partial, say that conclusions are based on partial evidence.

3. Start with high-level evidence.
   - Read `pseudoforge-corpus-overview.md` for clusters and high-signal functions.
   - Read `pseudoforge-corpus-index.json` for function tags, terms, caller/callee edges, imports, strings, warnings, buffer counts, and artifact paths.
   - Read `pseudoforge-corpus-metadata.json` for segments, imports, exports, strings, names, and global relationships.

4. Build a focused context pack when tools are available.

```powershell
python -B .\tools\pseudoforge_corpus_qa.py `
  "<corpus-root>\pseudoforge-corpus-index.json" `
  "<user question>" `
  --context-out "<corpus-root>\qa-context.md"
```

Use the generated `qa-context.md` as the primary evidence for the answer. If the question changes, regenerate a new context pack for the new question.

5. Retrieve manually when tools are unavailable.
   - Search `functions` entries in `pseudoforge-corpus-index.json`.
   - Rank matches by function name, tags, terms, imports, strings, interesting lines, and caller/callee names.
   - Expand one hop to callers and callees for behavior questions.
   - Read the selected function summary, cleaned pseudocode, raw pseudocode, diff, rename map, rule report, buffer contracts, and warnings before making nontrivial claims.

6. Answer with grounding.
   - Lead with the direct answer.
   - Cite evidence as `EA function_name artifact_path`.
   - Separate confirmed evidence from inference.
   - Say `unknown from the provided artifacts` when the evidence is insufficient.

## Retrieval Heuristics

For broad architecture questions:

1. Use `overview.top_clusters` and `clusters`.
2. Identify entrypoints, dispatchers, callbacks, framework setup, memory/pool code, registry/file code, and high-warning functions.
3. Summarize modules by evidence-backed function groups, not by guessed source files.

For function-specific questions:

1. Match by exact EA, function name, or substring.
2. Read the function's `.ida-batch-summary.json`.
3. Read `.cleaned.cpp`.
4. Compare with `.raw.cpp` and `.raw-vs-cleaned.diff` when the claim depends on transformed pseudocode.

For IOCTL or dispatch questions:

Search for tags and terms:

- `ioctl`
- `dispatch`
- `IRP_MJ_DEVICE_CONTROL`
- `IoGetCurrentIrpStackLocation`
- `DeviceIoControl`
- `IoControlCode`
- `case 0x`
- `buffer_contracts`

Then inspect buffer contract artifacts and caller/callee edges.

For callback questions:

Search for:

- `callback`
- `process_thread`
- `object_callback`
- `PsSetCreateProcessNotifyRoutine`
- `PsSetCreateThreadNotifyRoutine`
- `PsSetLoadImageNotifyRoutine`
- `ObRegisterCallbacks`
- `CmRegisterCallback`
- `FltRegisterFilter`

Then inspect registration functions, unregister/cleanup paths, and callback implementations.

For memory, pool, and buffer questions:

Search for:

- `memory`
- `pool`
- `Mm`
- `ExAllocatePool`
- `ExFreePool`
- `ProbeForRead`
- `ProbeForWrite`
- `MDL`
- `buffer_contracts`
- `warnings`

Treat buffer contracts as recovered evidence, then verify critical details against cleaned/raw pseudocode.

For API, import, or string questions:

1. Check `metadata.imports`, `metadata.exports`, `metadata.strings`, and `metadata.names`.
2. Use `imports_called` and `strings_referenced` in each function index item.
3. Expand to related functions through callers and callees.

For Korean questions, map common terms before retrieval:

- `프로세스`, `스레드`, `쓰레드` -> `process_thread`
- `콜백` -> `callback`
- `오브젝트`, `핸들` -> `object_callback`
- `메모리`, `풀`, `가상`, `물리` -> `memory`
- `아이오컨트롤`, `장치제어` -> `ioctl`
- `디스패치` -> `dispatch`
- `엔트리`, `드라이버엔트리` -> `entrypoint`
- `레지스트리`, `파일` -> `io_registry`

## Evidence Rules

- Do not invent missing source structure, file names, class names, or intent.
- Do not treat LLM rename status or generated names as authoritative by itself.
- Prefer `.cleaned.cpp` for readability, but verify risky claims with `.raw.cpp` and `.raw-vs-cleaned.diff`.
- Treat `.rename-map.json` as the rename decision record.
- Treat `.rule-report.json` as deterministic rule evidence, including rejected emissions and rule errors.
- Treat `.warnings.json` as review prompts, not automatically confirmed bugs.
- Treat `pseudoforge-corpus-metadata.json` as IDA-derived global evidence.
- Treat `pseudoforge-corpus-index.json` as a retrieval layer, not a replacement for reading detailed artifacts.
- If artifacts are missing or malformed, state which evidence is unavailable and continue with the remaining corpus.

## Recommended Answer Shape

Use this shape for most questions:

```markdown
<direct answer>

Evidence:
- `<EA>` `<function>`: <claim> ([artifact path])
- `<EA>` `<function>`: <claim> ([artifact path])

Inference:
- <clearly marked inference, if needed>

Unknowns / gaps:
- <what the corpus does not prove>
```

For architecture questions, use:

```markdown
Overall structure:
1. <subsystem or flow>
2. <subsystem or flow>
3. <subsystem or flow>

High-signal functions:
- `<EA>` `<function>`: <role> ([artifact path])

Call and data flow:
- <caller -> callee or registration -> callback relationship>

Gaps:
- <missing symbols, skipped functions, partial run limits, or unverified inference>
```

## Useful Commands

Rebuild an index from an existing output directory:

```powershell
python -B .\tools\pseudoforge_corpus_index.py "<corpus-root>"
```

Generate a focused context pack:

```powershell
python -B .\tools\pseudoforge_corpus_qa.py `
  "<corpus-root>\pseudoforge-corpus-index.json" `
  "<question>" `
  --context-out "<corpus-root>\qa-context.md"
```

Run deterministic retrieval without an LLM:

```powershell
python -B .\tools\pseudoforge_corpus_qa.py `
  "<corpus-root>\pseudoforge-corpus-index.json" `
  "<question>"
```

Search the corpus directly:

```powershell
rg -n "IRP_MJ_DEVICE_CONTROL|ObRegisterCallbacks|PsSet|Zw|ExAllocatePool" "<corpus-root>"
```

## Operational Boundaries

- Do not rerun IDA batch analysis unless the user explicitly asks to regenerate artifacts.
- Do not modify corpus artifacts unless the user asks for indexing, regeneration, or cleanup.
- Prefer writing temporary or focused context files over editing original bundles.
- When answering security-sensitive questions, stay grounded in defensive reverse engineering evidence and avoid operational bypass, evasion, or exploitation guidance.
